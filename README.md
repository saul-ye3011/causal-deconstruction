# 因果解构 Causal Deconstruction

一套让 AI 从「回答问题」切换到「拆解问题」的思维协议。

---

## 它解决什么问题

多数 AI 对话存在一个结构性缺陷：

用户带着当前认知提问 → AI 回答表面问题 → 真正的问题从未被触及

这个循环的根源在于，用户的提问本身就被自身的盲区塑形了。表述越流畅，越容易掩盖真实意图与实际提问之间的偏差。

因果解构协议打破这个循环。它让 AI 先校准问题本身，再构建可追踪的因果链，最后收束为精炼的答案。

## 它是什么

一份 System Prompt。将它注入任何大语言模型，模型的响应模式会从「检索式应答」重构为「因果推理引擎」。

它定义了一套固定的解构路径：

```
校准（问对问题）→ 识别变量（找到真正起作用的因子）→ 视角转向（找到让问题变简单的角度）→ 追溯因果（构建可验证的逻辑链）→ 精炼答案（一针见血）
```

以及一套极简符号语言，让因果关系可视化：

| 符号 | 含义 |
|------|------|
| `→` | 单向因果（A 导致 B） |
| `↔` | 相互影响 |
| `↑` `↓` | 提升 / 下降 |
| `≠` | 对立关系 |
| `≈` | 近似但不完全相同 |
| `∑` | 多因素汇总 |
| `★` | 关键洞察 |

## 实际效果

**提问：** 为什么行动优于犹豫？

**逻辑链：**

```
信息不足 → 决策准确性↓
犹豫 → 停滞 → 信息静止 → 信息持续不足 → 陷入决策困境
行动 → 开拓 → 信息更新 → 信息量↑ → 决策准确性↑
★ 犹豫 ≠ 谨慎思考，这是信息获取的停滞
```

**答案：** 因为行动本身就是获取信息的过程，而信息越充分，决策越明朗。

每一步都可追踪、可验证、可反驳。没有「我认为」，没有「通常来说」，只有因果链条的自然收束。

## 适用场景

**决策卡点** 选项越想越多，却越来越不清楚该选哪个。问题往往在于有隐藏变量没有被识别出来。因果解构会把那些被情绪、习惯、惯性遮蔽的因子暴露出来。

**问题诊断** 症状和根因之间隔着好几层。无论是代码 bug、业务瓶颈还是人际摩擦，因果链会把中间的传导路径一层层铺开。

**认知校准** 你问的和你真正想知道的可能是两件事。因果解构的第零步「校准」会在回答之前先检验问题本身的准确性。

**学习加速** 理解一件事的因果结构，比记忆它的表面特征高效得多。因果链让知识从「知道」变成「理解」。

**论证验证** 检验一条推理链是否真的成立。当检测到逻辑漏洞、未经验证的假设、因果强行关联时，因果解构会立即中断并指出问题。

## 使用方法

1. 复制 [`prompt.md`](./prompt.md) 的全部内容
2. 粘贴到你使用的 AI 对话工具的 System Prompt / 自定义指令中
3. 开始提问

兼容所有主流大语言模型（Claude、GPT、Gemini、DeepSeek 等）。模型能力越强，因果链的深度和准确性越高。

### 进阶用法

每次解构完成后，你可以选择两个方向继续探索：

- **→ 泛化** 拓宽视野，看见相关领域的启发
- **→ 进阶** 深入剖析，逐层拆解每个变量的内在机制

## 它的边界

因果解构擅长处理有因果结构的问题。以下场景它帮不了你：

- 纯粹的情感陪伴（它会分析情绪的成因，但不会扮演倾听者）
- 没有逻辑结构的创意发散（它会给创意一个骨架，但不会天马行空）
- 需要最新数据的事实查询（它处理逻辑，不处理信息检索）

## 设计哲学

**「多数问题被处理了两次：第一次是表面响应，第二次才是真正理解。因果解构跳过第一次，直抵第二次。」**

这个协议的每一条规则都服务于同一个目标：让思维的轨迹可被追踪。

逻辑链是思维的 X 光片。符号是辅助理解的工具，当自然语言更清晰时，不强行使用符号。答案是逻辑链的自然收束，而非跳跃式结论。

它不迎合预设立场，不为错误的前提背书，不为有偏差的追问继续推演。它只忠于逻辑本身。
